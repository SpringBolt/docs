### Introduction

The **Binary Search** pattern is used to find a target value in a **sorted** search space by repeatedly cutting the space in half.

Instead of scanning linearly, we compare with the middle element and eliminate half of the remaining candidates at each step.

This gives logarithmic performance: **$O(\log n)$**.

### Intuition: "Eliminate Half Every Time"

Given sorted array `[1, 3, 5, 7, 9, 11]`, target `7`:

1. Check middle (`5`) -> target is bigger.
2. Ignore left half.
3. Check middle of right half (`9`) -> target is smaller.
4. Ignore right part.
5. Land on `7`.

Each comparison discards about 50% of the remaining search range.

---

### Core Variations

#### 1. Exact Match

* Find index where `arr[mid] == target`.
* Return `-1` if not found.

#### 2. First/Last Occurrence

* Continue searching left/right after a match.
* Useful in duplicate-heavy arrays.

#### 3. Search on Answer Space

* Binary search over possible answers (capacity, speed, threshold).
* Use feasibility check to move boundaries.

### Implementation Examples

#### Example A: Classic Binary Search
**Problem:** Find target index in a sorted integer array.

```java icon="code" title="Binary Search - Exact Match"
public int search(int[] nums, int target) {
    int left = 0;
    int right = nums.length - 1;

    while (left <= right) {
        int mid = left + (right - left) / 2;

        if (nums[mid] == target) {
            return mid;
        } else if (nums[mid] < target) {
            left = mid + 1;
        } else {
            right = mid - 1;
        }
    }

    return -1;
}
```

#### Example B: First Occurrence in Sorted Array

**Problem:** Return first index of target in sorted array with duplicates.

```java icon="code" title="Binary Search - First Occurrence"
public int firstOccurrence(int[] nums, int target) {
    int left = 0;
    int right = nums.length - 1;
    int answer = -1;

    while (left <= right) {
        int mid = left + (right - left) / 2;

        if (nums[mid] == target) {
            answer = mid;
            right = mid - 1; // keep searching on left
        } else if (nums[mid] < target) {
            left = mid + 1;
        } else {
            right = mid - 1;
        }
    }

    return answer;
}
```

### Real-World Analogy

Think of looking up a word in a physical dictionary.

* You open somewhere in the middle.
* Based on alphabetical order, you ignore half of the pages.
* Repeat until the exact word is found.

That is Binary Search in daily life.

---

### When to Use

Look for these clues:

* Input is **sorted** or monotonic.
* Need fast lookup on large search space.
* Terms like **first true**, **minimum feasible**, **lower bound**.

### Pros & Cons

| Pros | Cons |
| --- | --- |
| **Very Fast:** $O(\log n)$ lookups. | **Requires Order:** Needs sorted/monotonic structure. |
| **Scalable:** Works well for huge arrays/ranges. | **Boundary Bugs:** Off-by-one errors are common. |
| **General Pattern:** Also works on answer space. | **No Benefit on Tiny Inputs:** Linear scan may be simpler. |

### Summary Table

| Variant | Time | Space | Typical Use |
| --- | --- | --- | --- |
| **Exact Match** | $O(\log n)$ | $O(1)$ | Search element in sorted array |
| **First/Last Occurrence** | $O(\log n)$ | $O(1)$ | Duplicates, range boundaries |
| **Answer-Space Search** | $O(\log R)$ | $O(1)$ | Capacity/speed threshold problems |
