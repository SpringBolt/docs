### Introduction

The **Merge Intervals** pattern handles problems involving ranges like `[start, end]`, especially when overlap detection or timeline compression is required.

The classic flow is:

1. Sort by start time.
2. Scan intervals in order.
3. Merge if current start is within previous end.

### Intuition

Brute force compares many pairs of intervals:

```java icon="code" title="Brute Force Overlap Check"
// Brute Force: O(n^2)
for (int i = 0; i < intervals.length; i++) {
    for (int j = i + 1; j < intervals.length; j++) {
        // Check overlap between intervals[i] and intervals[j]
    }
}
```

After sorting, each interval only needs comparison with the **last merged interval**, which removes redundant checks.

### What are Predictable Dynamics?

Sorting by start creates a stable order:

* If `next.start <= current.end`, overlap exists and we can extend `current.end`.
* If `next.start > current.end`, no future interval can overlap with `current` (because starts only increase).

This monotonic start ordering enables a linear merge pass after sorting.

### Core Strategies

#### 1. Standard Merge

* Merge all overlapping intervals into disjoint blocks.
* Use Case: schedule consolidation.

#### 2. Insert and Merge

* Insert one interval into sorted non-overlapping list.
* Merge local overlaps and keep final ordering.

#### 3. Interval Intersection

* Two sorted interval lists.
* Two pointers produce overlap segments.

### Implementation Examples

#### Example A: Merge Overlapping Intervals

```java icon="code" title="Intervals - Merge"
public int[][] merge(int[][] intervals) {
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));
    List<int[]> merged = new ArrayList<>();

    for (int[] interval : intervals) {
        if (merged.isEmpty() || merged.get(merged.size() - 1)[1] < interval[0]) {
            merged.add(new int[] { interval[0], interval[1] });
        } else {
            int[] last = merged.get(merged.size() - 1);
            last[1] = Math.max(last[1], interval[1]);
        }
    }

    return merged.toArray(new int[merged.size()][]);
}
```

#### Example B: Interval Intersection (Two Pointers)

```java icon="code" title="Intervals - Intersection"
public int[][] intervalIntersection(int[][] first, int[][] second) {
    List<int[]> result = new ArrayList<>();
    int i = 0, j = 0;

    while (i < first.length && j < second.length) {
        int start = Math.max(first[i][0], second[j][0]);
        int end = Math.min(first[i][1], second[j][1]);

        if (start <= end) {
            result.add(new int[] { start, end });
        }

        if (first[i][1] < second[j][1]) i++;
        else j++;
    }

    return result.toArray(new int[result.size()][]);
}
```


### Real-World Analogies

#### 1. Conceptual Analogy: Calendar Consolidation

Back-to-back overlapping meetings become one blocked time segment so your calendar reflects actual unavailable ranges.

#### 2. Technical Analogy: OS CPU Busy-Window Coalescing

A scheduler can merge overlapping CPU busy intervals to compute true utilization windows without double-counting overlap.

### When to Use (and How to Recognize It)

* Input is list of ranges/time spans.
* Terms: overlap, merge, schedule, conflict, interval.
* Need condensed disjoint ranges.
* Sorting then single pass feels natural.

### Pros & Cons

| Pros | Cons |
| --- | --- |
| Clean template for many interval tasks. | Usually requires sorting: **O(n log n)**. |
| Single scan after sorting. | Mutating intervals can cause side effects if reused elsewhere. |
| Easy to adapt to insert/intersection variants. | Requires careful boundary handling (`<=` vs `<`). |
