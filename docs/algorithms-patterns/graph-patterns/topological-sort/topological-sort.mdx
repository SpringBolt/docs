### Introduction

The **Topological Sort** pattern generates an ordering of nodes in a **Directed Acyclic Graph (DAG)** such that for every edge `u -> v`, node `u` appears before `v`.

It is the standard approach for dependency-driven tasks like course scheduling, build pipelines, and job execution plans.

### Intuition

Brute force tries permutations and validates dependencies, which is intractable.

```java icon="code" title="Brute Force Dependency Ordering"
// Extremely expensive idea: test all permutations.
// for each permutation, check whether every edge u -> v
// satisfies position(u) < position(v)
```

Topological Sort instead repeatedly picks nodes with no remaining unmet prerequisites.

### What are Predictable Dynamics?

In a DAG:

* At least one node has in-degree `0`.
* Removing such a node and its outgoing edges may create new in-degree `0` nodes.
* If no in-degree `0` node exists before processing all nodes, a cycle must exist.

This dependency-release dynamic drives Kahn’s BFS approach.

### Core Strategies

#### 1. Kahn’s Algorithm (BFS + In-Degree)

* Queue all nodes with in-degree `0`.
* Pop, record, decrement neighbors’ in-degree.
* If processed count < total nodes, cycle exists.

#### 2. DFS Postorder Topological Sort

* DFS each node.
* Push node after exploring descendants.
* Reverse finishing order gives topological order.

#### 3. Lexicographically Smallest Topological Order

* Same as Kahn but use min-heap instead of queue.
* Always picks smallest available node.

### Implementation Examples

#### Example A: BFS Traversal on Graph

```java icon="code" title="Graph - BFS Traversal"
public List<Integer> bfs(int n, List<List<Integer>> graph, int source) {
    boolean[] visited = new boolean[n];
    Queue<Integer> queue = new LinkedList<>();
    List<Integer> order = new ArrayList<>();

    queue.offer(source);
    visited[source] = true;

    while (!queue.isEmpty()) {
        int node = queue.poll();
        order.add(node);

        for (int neighbor : graph.get(node)) {
            if (!visited[neighbor]) {
                visited[neighbor] = true;
                queue.offer(neighbor);
            }
        }
    }

    return order;
}
```

#### Example B: DFS Traversal with Recursion

```java icon="code" title="Graph - DFS Traversal"
public void dfs(int node, List<List<Integer>> graph, boolean[] visited, List<Integer> order) {
    visited[node] = true;
    order.add(node);

    for (int neighbor : graph.get(node)) {
        if (!visited[neighbor]) {
            dfs(neighbor, graph, visited, order);
        }
    }
}
```


### Real-World Analogies

#### 1. Conceptual Analogy: University Course Plan

You can register only for courses whose prerequisites are complete. As you finish courses, new courses become available.

#### 2. Technical Analogy: Build System Dependency Graph

A build tool compiles modules with no pending dependencies first. Each completed module unlocks downstream modules until all buildable targets are processed.

### When to Use (and How to Recognize It)

* Directed dependency relation: “A must happen before B”.
* Need valid execution/order plan.
* Need cycle detection in prerequisite graph.
* Terms: dependency, prerequisite, ordering, scheduling.

### Pros & Cons

| Pros | Cons |
| --- | --- |
| Handles ordering and cycle detection together. | Only valid for DAGs (cycles break full ordering). |
| Linear complexity: **O(V + E)** for BFS/DFS variants. | Multiple valid orders may exist; result is not unique. |
| Extensible to deterministic ordering via heap. | Requires graph and in-degree bookkeeping. |
