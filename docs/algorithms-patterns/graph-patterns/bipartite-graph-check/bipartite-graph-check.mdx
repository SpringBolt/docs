### Introduction

The **Bipartite Graph Check** pattern verifies whether graph nodes can be split into two sets such that every edge connects one set to the other.

This pattern appears in team partitioning, scheduling constraints, and odd-cycle detection.

### Intuition: "Two-Color the Graph"

If a graph is bipartite, we can color each node with one of two colors (`0` or `1`) so adjacent nodes always have opposite colors.

If we ever find an edge connecting two same-colored nodes, graph is not bipartite.

Both BFS and DFS coloring work.

---

### Core Variations

#### 1. BFS Two-Coloring

* Level-style coloring with queue.
* Good iterative template.

#### 2. DFS Two-Coloring

* Recursive coloring and conflict checking.
* Compact and clean.

#### 3. Odd Cycle Detection

* Undirected graph is bipartite iff it has no odd-length cycle.

### Implementation Examples

#### Example A: BFS Bipartite Check
**Problem:** Return true if undirected graph is bipartite.

```java icon="code" title="Bipartite - BFS Coloring"
public boolean isBipartiteBFS(List<List<Integer>> graph) {
    int n = graph.size();
    int[] color = new int[n];
    Arrays.fill(color, -1); // -1 = uncolored

    for (int start = 0; start < n; start++) {
        if (color[start] != -1) continue;

        Queue<Integer> queue = new LinkedList<>();
        queue.offer(start);
        color[start] = 0;

        while (!queue.isEmpty()) {
            int node = queue.poll();

            for (int neighbor : graph.get(node)) {
                if (color[neighbor] == -1) {
                    color[neighbor] = 1 - color[node];
                    queue.offer(neighbor);
                } else if (color[neighbor] == color[node]) {
                    return false;
                }
            }
        }
    }

    return true;
}
```

#### Example B: DFS Bipartite Check

**Problem:** Return true if graph is bipartite using DFS.

```java icon="code" title="Bipartite - DFS Coloring"
public boolean isBipartiteDFS(List<List<Integer>> graph) {
    int n = graph.size();
    int[] color = new int[n];
    Arrays.fill(color, -1);

    for (int i = 0; i < n; i++) {
        if (color[i] == -1 && !dfsColor(i, 0, graph, color)) {
            return false;
        }
    }

    return true;
}

private boolean dfsColor(int node, int currentColor, List<List<Integer>> graph, int[] color) {
    color[node] = currentColor;

    for (int neighbor : graph.get(node)) {
        if (color[neighbor] == -1) {
            if (!dfsColor(neighbor, 1 - currentColor, graph, color)) return false;
        } else if (color[neighbor] == currentColor) {
            return false;
        }
    }

    return true;
}
```

### Real-World Analogy

Think of assigning people to two teams where rivals cannot be in same team.

* If assignments can be completed without conflict, graph is bipartite.
* Any unavoidable odd rivalry loop breaks feasibility.

---

### When to Use

Look for these clues:

* Need to divide entities into two valid groups.
* Constraints expressed as "must be in opposite groups".
* Detect odd cycle in undirected graph.

### Pros & Cons

| Pros | Cons |
| --- | --- |
| **Simple Coloring Model:** Easy to implement and reason about. | **Only Binary Partition:** Does not solve general graph coloring. |
| **Linear Complexity:** Efficient for large sparse graphs. | **Disconnected Graph Handling Required:** Must start BFS/DFS from each component. |
| **Useful Structural Check:** Quickly reveals odd-cycle conflicts. | **Directed Interpretation Varies:** Standard form is undirected. |
