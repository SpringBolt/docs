### Introduction

The **Queue Using Two Stacks** pattern implements FIFO behavior using two LIFO stacks.

It is a classic data-structure transformation problem that demonstrates amortized analysis.

Even though individual operations may sometimes move many elements, average cost per operation remains constant.

### Intuition: "Reverse Twice to Restore FIFO"

A stack reverses order.

If we push enqueue elements into `inStack`, newest stays on top.

When dequeue is needed:

* Move all elements from `inStack` to `outStack` (first reverse).
* Top of `outStack` becomes the oldest element (FIFO order restored).

This transfer happens only when `outStack` is empty.

---

### Core Variations

#### 1. Lazy Transfer (Amortized O(1))

* Transfer only when dequeue/peek requires it.
* Best practical approach.

#### 2. Eager Transfer

* Transfer on every enqueue.
* Simpler dequeue but slower enqueue.

#### 3. Queue Using Two Deques

* Similar logic with custom interfaces.
* Useful in restricted API settings.

---

### Implementation Examples

#### Example A: Amortized O(1) Queue
**Problem:** Implement queue with stacks supporting `push`, `pop`, `peek`, and `empty`.

```java icon="code" title="Queue - Using Two Stacks"
class MyQueue {
    private Deque<Integer> inStack;
    private Deque<Integer> outStack;

    public MyQueue() {
        inStack = new ArrayDeque<>();
        outStack = new ArrayDeque<>();
    }

    public void push(int x) {
        inStack.push(x);
    }

    public int pop() {
        moveIfNeeded();
        return outStack.pop();
    }

    public int peek() {
        moveIfNeeded();
        return outStack.peek();
    }

    public boolean empty() {
        return inStack.isEmpty() && outStack.isEmpty();
    }

    private void moveIfNeeded() {
        if (outStack.isEmpty()) {
            while (!inStack.isEmpty()) {
                outStack.push(inStack.pop());
            }
        }
    }
}
```

#### Example B: Eager Transfer Strategy

**Problem:** Keep front always ready by transferring during `push`.

```java icon="code" title="Queue - Two Stacks (Eager Transfer)"
class MyQueue {
    private Deque<Integer> main;
    private Deque<Integer> helper;

    public MyQueue() {
        main = new ArrayDeque<>();
        helper = new ArrayDeque<>();
    }

    public void push(int x) {
        while (!main.isEmpty()) {
            helper.push(main.pop());
        }

        main.push(x);

        while (!helper.isEmpty()) {
            main.push(helper.pop());
        }
    }

    public int pop() {
        return main.pop();
    }

    public int peek() {
        return main.peek();
    }

    public boolean empty() {
        return main.isEmpty();
    }
}
```

---

### Real-World Analogy

Imagine two stacks of trays:

* New trays are stacked in storage stack.
* When serving starts, trays are moved to serving stack.
* Serving stack now gives oldest tray first.

You avoid repeated movement by transferring only when serving stack becomes empty.

---

### When to Use

Look for these clues:

* Need queue behavior but only stack operations available.
* Interview asks for API emulation between data structures.
* Amortized complexity discussion is expected.

### Pros & Cons

| Pros | Cons |
| --- | --- |
| **Elegant Transformation:** Demonstrates stack/queue duality. | **Worst-Case Single Operation:** One pop can trigger full transfer. |
| **Amortized Efficiency:** Average `O(1)` for core operations. | **Extra Memory:** Uses two stacks. |
| **Common Interview Pattern:** Frequently tested design exercise. | **Less Direct than Native Queue:** More helper logic required. |
